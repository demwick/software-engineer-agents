# State file layout — v2.0.0 audit

**Last updated:** 2026-04-15
**Companion to:** `docs/specs/2026-04-15-scope-and-state-refactor.md`
**Status:** Design-only audit (Phase 2). Consolidation is executed in Phase 6, gated on user sign-off of the "Known simplification opportunities" section below.

This document is the canonical reference for every file the plugin writes inside a project's `.sea/` directory (plus the per-agent `MEMORY.md` files the platform manages outside `.sea/`). For each file it records the writer(s), reader(s), format, required fields, behavior on missing or corrupted state, and how it can be recreated.

Phase 2 of the scope-cut refactor produced this audit. The v1.0.0 reference document is preserved verbatim at the bottom of this file under "v1.0.0 reference (historical)".

---

## Scope

Two storage layers are in play:

1. **Project runtime state** — under `<project-root>/.sea/`. Gitignored by default. Regenerated by `/sea-init` if lost. Contains the roadmap, per-phase plans and progress, transient Stop-hook markers, and ad-hoc outputs from the supplementary commands (`/sea-review`, `/sea-debug`, `/sea-ship`, `/sea-diagnose`).
2. **Per-agent cross-session memory** — under `.claude/agent-memory/<agent>/MEMORY.md`. Platform-managed via the subagent `memory: project` frontmatter field. Not gitignored by default; commit or ignore per project preference.

The per-file detail section below covers both layers. The cross-file invariants apply to layer 1 only — the platform owns layer 1 invariants for `MEMORY.md`.

## Convention

Paths below use `<project>` for the project root, `N` for a phase number, `<N>` for a debug session index (also an integer), and `<timestamp>` / `<YYYYMMDD>` for ISO-like timestamp suffixes the writing code generates at runtime.

## Per-file detail

The inventory table above is the index. The per-file sections below answer four questions for each entry:

- **Writer(s)** — exact skill, agent, hook, or script that writes it, with file paths.
- **Reader(s)** — every consumer, with file paths.
- **What happens if it is missing?** — does the plugin degrade, recover, or crash?
- **What happens if it is corrupted?** — who detects, how does recovery work?

### `state.json`

- **Writer(s):**
  - `skills/sea-init/SKILL.md` (initial `Write`, the only place raw `Write` is permitted on this file).
  - `scripts/state-update.sh` (every subsequent update — atomic `jq`-based merge via a temp file, `mv`-into-place, preserves required fields, validates schema after merge). Called from `skills/sea-go/SKILL.md:102`, `skills/sea-quick/SKILL.md`, `skills/sea-milestone/SKILL.md`, `skills/sea-roadmap/SKILL.md`.
  - `hooks/state-tracker` (PostToolUse) refreshes `last_edit` via the same helper.
- **Reader(s):**
  - `skills/sea-status/SKILL.md:25-37`, `skills/sea-go/SKILL.md:24`, `skills/sea-quick/SKILL.md`, `skills/sea-milestone/SKILL.md:24`, `skills/sea-diagnose/SKILL.md:92`, `skills/sea-undo/SKILL.md`.
  - `hooks/session-start` (SessionStart) reads on every session launch for the `additionalContext` payload.
- **Missing:** the plugin treats this as "uninitialized project" — `/sea-go`, `/sea-quick`, `/sea-status`, `/sea-milestone` all halt and route the user to `/sea-init`. `SessionStart` hook injects an empty `additionalContext`.
- **Corrupted:** `scripts/state-update.sh` fails closed. `jq` parse error → exit 4, merge aborted, the existing file is left untouched. If the *on-disk* file is corrupted (hand-edited into invalid JSON), the next `state-update` call refuses to merge and the user must restore from git or re-run `/sea-init` with `--fresh`.

### `roadmap.md`

- **Writer(s):** `/sea-init` (initial, via `planner` Mode A); `/sea-roadmap` (edits); `/sea-go` (flips Phase N status to `done` at phase completion, see `skills/sea-go/SKILL.md:109`); `/sea-milestone` (appends new phases below a milestone boundary marker).
- **Reader(s):** `/sea-status` (phase count), `/sea-go` (next-pending-phase pick), `/sea-quick` (consistency check), `/sea-milestone`, `/sea-review` (phase-range lookup), `hooks/session-start` (short state summary in context).
- **Missing:** same as `state.json` — project is "uninitialized". `/sea-roadmap` specifically refuses to create it; only `/sea-init` does.
- **Corrupted:** no automated detection. Readers parse what they can; `/sea-status` will show a degraded count. Hand-editing is *expected*, so schema drift is mostly tolerated — only `### Phase N:` headings and `**Status:**` lines are load-bearing.

### `diagnose.json`

- **Writer(s):** `skills/sea-diagnose/SKILL.md:79` writes the full report after the researcher agent completes its audit.
- **Reader(s):** `skills/sea-status/SKILL.md:41` (header), `skills/sea-quick/SKILL.md:28-93` (suggests re-running `/sea-diagnose` after a fix if the task came from `priority_actions`).
- **Missing:** expected — `/sea-status` simply omits the "last audit" line; `/sea-quick` skips the post-fix reminder.
- **Corrupted:** no automated detection. `/sea-status` falls back to "last audit: unparseable" and moves on. No other caller hard-depends on structure.

### `.needs-verify`

- **Writer(s):**
  - `skills/sea-go/SKILL.md:80` and `skills/sea-quick/SKILL.md:78` create it with content `0` after the executor finishes a task or phase.
  - `hooks/auto-qa:94` rewrites it with an incremented attempt count on a failed retry.
  - `hooks/auto-qa` deletes it on success, on loop-protection give-up, on host-compat failure, and when no test runner is detected.
- **Reader(s):** `hooks/auto-qa:21,30` (existence check + integer content as retry counter), plus the companion reference doc `skills/sea-go/references/auto-qa-protocol.md`.
- **Missing:** this is the *normal* state. Its presence means "the executor just finished writing code; the Stop hook should run tests". No caller crashes on absence.
- **Corrupted:** `hooks/auto-qa:30-31` treats a missing or non-integer content as `0` (defensive default). A marker containing `"abc"` will be silently treated as a first-attempt retry. This is acceptable for a transient marker but is one of the smells flagged in the refactor review — see Known simplification opportunities.

### `.last-verify.log`

- **Writer(s):** `hooks/auto-qa:57,60,69` (redirects test command stdout/stderr, appends host-compat trailer).
- **Reader(s):** `skills/sea-status/SKILL.md:43` (surfaces mtime + tail), `skills/sea-debug/SKILL.md:28,47` (error bait for triage), `tests/run-tests.sh:191` (regression assertion on host-compat warning content).
- **Missing:** expected before the first Stop-hook run. `/sea-status` omits the last-run line; `/sea-debug` skips the "last auto-QA run failed X minutes ago" nudge.
- **Corrupted:** not possible in the usual sense — the file is a raw dump of another process's output. Consumers only read it as opaque text (mtime, last ~30 lines, substring match).

### `phases/phase-N/plan.md`

- **Writer(s):** `planner` agent in Mode B (Phase Planning), called by `skills/sea-go/SKILL.md:41`. The planner is the only writer.
- **Reader(s):** `executor` agent (`agents/executor.md:28`) for task-by-task execution; `skills/sea-go/SKILL.md:38` (plan-presence check, re-reads on resume); `skills/sea-status/SKILL.md:28` (optional active-phase display).
- **Missing:** `skills/sea-go/SKILL.md:41` treats "no plan" as a signal to launch the planner, so missing-by-design. Missing on a phase the planner previously wrote is an anomaly — `/sea-go` will regenerate it, losing any `[[ ASK: ... ]]` decisions the user already answered.
- **Corrupted:** no automated validation. The executor is tolerant of minor structure drift but will surface `_common.md:2` "Manage Confusion Actively" if the file is so broken it cannot identify tasks, and stop the line.

### `phases/phase-N/progress.json`

- **Writer(s):** `agents/executor.md:34,39,54-58` — one atomic write after each successful task commit inside a phase.
- **Reader(s):** `skills/sea-go/SKILL.md:59` (resume decision); `agents/executor.md:29` (skip completed tasks); `skills/sea-debug/SKILL.md:29` (detect blocked phase); `skills/sea-undo/SKILL.md:31,89` (fallback when summary is missing, plus deletion on revert).
- **Missing:** normal for a fresh phase (nothing started yet) or a completed phase (deleted at phase end). `/sea-go` interprets "missing" as "start at task 1".
- **Corrupted:** `/sea-go` and the executor refuse to parse non-JSON and fall back to "start at task 1". Silent data loss risk: if `completed_tasks[]` is lost, the executor re-runs tasks — benign because each task is an atomic, idempotent commit-or-skip, but worth flagging.

### `phases/phase-N/summary.md`

- **Writer(s):** `skills/sea-go/SKILL.md:110` writes the summary when the phase completes successfully.
- **Reader(s):** `skills/sea-status/SKILL.md` (historical display), `skills/sea-review/SKILL.md:33` (phase-range pickup), `skills/sea-undo/SKILL.md:30` (commit range extraction).
- **Missing:** the phase is treated as "not yet completed" by every reader. `/sea-undo` falls back to `progress.json` + `git log`. `/sea-review` cannot target the phase.
- **Corrupted:** `/sea-undo`'s commit-range regex fails closed and surfaces an error rather than attempting a wrong revert.

### `phases/phase-N/summary.md.reverted-<timestamp>`

- **Writer(s):** `skills/sea-undo/SKILL.md:88` renames the original `summary.md` to this path when the phase is reverted — never deletes history.
- **Reader(s):** humans only. No code reads this file.
- **Missing:** expected in every unreverted phase.
- **Corrupted:** not possible — it is a rename, so the file is whatever `summary.md` was at the moment of the revert.

### `phases/phase-N/review.md`

- **Writer(s):** `agents/reviewer.md:86,94` writes the full markdown report from `/sea-review` or `/sea-go` Step 6.5 (on medium/complex phases after auto-QA passes).
- **Reader(s):** humans; `/sea-review` re-reads for repeated runs.
- **Missing:** expected for phases that have not been reviewed. No caller crashes on absence.
- **Corrupted:** no automated detection. Humans re-run `/sea-review` to overwrite.

### `reviews/ad-hoc-<timestamp>.md`

- **Writer(s):** `agents/reviewer.md` (via `/sea-review <range>` for range-based, non-phase reviews).
- **Reader(s):** humans only.
- **Missing:** expected by default — ad-hoc reviews are opt-in.
- **Corrupted:** humans re-run.

### `phases/archived-<YYYYMMDD>-phase-N/`

- **Writer(s):** `skills/sea-roadmap/SKILL.md` and `skills/sea-init/SKILL.md` rename an old `phases/phase-N/` directory to this form when the phase is removed or the project is re-scaffolded. Spec design: never delete, always archive.
- **Reader(s):** humans only.
- **Missing:** expected in projects that have never had a phase removed.
- **Corrupted:** not possible — archival is a directory rename.

### `debug/session-<N>/*.md`

- **Writer(s):** `agents/debugger.md:31,154` writes `reproduction.md`, `root-cause.md`, or `experiments.md` depending on triage outcome.
- **Reader(s):** humans; `skills/sea-debug/SKILL.md:36` increments `<N>` by counting existing session directories.
- **Missing:** expected — debug sessions are opt-in.
- **Corrupted:** session directories are append-only; humans re-run `/sea-debug` to produce a new session.

### `ship-report.json` and `ship/<category>.log`

- **Writer(s):** `skills/sea-ship/SKILL.md:59,101` writes the overall report and per-category logs after running the pre-merge quality gate.
- **Reader(s):** humans, optionally CI.
- **Missing:** expected until the first `/sea-ship` run.
- **Corrupted:** humans re-run `/sea-ship`.

### `.claude/agent-memory/<agent>/MEMORY.md`

- **Writer(s):** the named subagent, via the platform's `memory: project` mechanism. The plugin itself never touches this path.
- **Reader(s):** the same subagent on its next invocation. The platform auto-loads up to 25 KB.
- **Missing:** expected on first invocation; platform creates on demand.
- **Corrupted:** the platform handles recovery. SEA does not own this invariant.

## Cross-file invariants

These rules must hold across multiple state files. They are not enforced by a single validator — each reader relies on its own narrow check — but they are the assumptions the plugin's design rests on. A violation is a bug.

1. **`state.json.current_phase` is in-range.** `0 ≤ current_phase ≤ total_phases`. `total_phases` equals the count of `### Phase N:` sections in `roadmap.md`. Violation → `/sea-go` either picks a non-existent phase or refuses to advance.

2. **Active progress implies the active phase.** If `.sea/phases/phase-N/progress.json` exists, then `state.json.current_phase == N`. Put differently: there is at most one phase with an in-flight `progress.json`. Violation → executor and `/sea-go` disagree about which phase to resume.

3. **Summary exists only for completed phases.** `.sea/phases/phase-N/summary.md` exists if and only if `roadmap.md` marks Phase N as `done` (or `/sea-undo` has reverted the phase, in which case a `.reverted-<timestamp>` sibling replaces it). Violation → `/sea-status`, `/sea-review`, `/sea-undo` produce inconsistent views.

4. **`schema_version` is monotonic.** `state.json.schema_version` never decreases. All migrations are one-way. `scripts/state-update.sh` does not support downgrades; the `pre-scope-cut` git tag is the only rollback floor.

5. **Required `state.json` fields survive every update.** `schema_version`, `mode`, `created`, `current_phase`, `total_phases` are all always present after any `scripts/state-update.sh` call (exit 4 if the merge drops one). Violation can only happen via a raw `Write`/`Edit` on an existing `state.json`, which is explicitly forbidden in `CLAUDE.md:104`.

6. **`.needs-verify` existence is session-bounded.** If `.sea/.needs-verify` is present at the start of a new Claude Code session, either the previous session crashed before `hooks/auto-qa` could clear it, or the last `auto-qa` run hit loop-protection and gave up. In either case the presence represents "unfinished verification work" and `/sea-status` should surface it. (In practice no current reader does — this is a known follow-up.)

7. **`progress.json` self-consistency.** `progress.json.phase == N` for the containing `phases/phase-N/` directory, and `progress.json.current_task` > every element of `progress.json.completed_tasks[]`. Violation → executor re-runs tasks unnecessarily (benign) or skips unfinished tasks (not benign).

8. **Phase archival is append-only.** A directory renamed to `phases/archived-<YYYYMMDD>-phase-N/` is never reopened. `/sea-roadmap` refuses to resurrect archived phases; the user must re-add as a new phase number.

### What happens if `state.json` and `roadmap.md` disagree?

The most common real-world case this spec must answer.

| Disagreement | Detected by | Effect | Recovery |
|---|---|---|---|
| `state.json.current_phase > total_phases_in_roadmap` | `/sea-go` (picks a phase that does not exist in `roadmap.md`) | `/sea-go` halts with "phase N not found in roadmap" | User hand-edits `roadmap.md` to add the missing phase, or runs `/sea-roadmap` |
| `state.json.current_phase` points to a phase whose `roadmap.md` status is `done` | `/sea-go` (refuses to rerun a completed phase) | `/sea-go` halts with "Phase N is already done" | User bumps `current_phase` via `/sea-roadmap` or deletes the stale entry |
| `state.json.current_phase` points to a phase whose `roadmap.md` status is `pending`, but `phases/phase-N/summary.md` exists | `/sea-go` (inconsistency between summary and roadmap status) | Invariant 3 violated — `/sea-go` halts and surfaces `_common.md:2` "Manage Confusion Actively" | User either flips roadmap status to `done` or archives the stray summary |
| `total_phases` in `state.json` ≠ count of `### Phase N:` in `roadmap.md` | `/sea-status` (phase count mismatch) | `/sea-status` shows the roadmap count as authoritative and warns the user | User runs `/sea-roadmap` to re-sync, or edits `state.json` via `scripts/state-update.sh total_phases=K` |

In every case the plugin **stops the line rather than guessing**, per `_common.md:5`. There is no auto-reconciliation. This is deliberate: silently trusting one file over the other is the path to data corruption. Phase 6 may add a single-purpose validator skill to detect these drifts proactively, but it is not in scope for the current refactor.

# State File Layout

This document describes what software-engineer-agent writes inside a project's `.sea/` directory and how each file is used. You don't need to create any of these by hand — `/sea-init` generates them. This doc is the reference when you want to inspect, tweak, or debug state.

## Directory Layout

```
<project-root>/
└── .sea/                # added to .gitignore automatically by /sea-init
    ├── state.json               # canonical runtime state
    ├── roadmap.md               # human-readable phase list
    ├── diagnose.json            # latest health report (if /sea-diagnose has run)
    ├── .needs-verify            # transient marker — triggers the auto-QA hook
    ├── .last-verify.log         # last test-runner output (written by auto-qa)
    └── phases/
        ├── phase-1/
        │   ├── plan.md          # planner's output for this phase
        │   └── summary.md       # executor's completion note
        ├── phase-2/
        │   ├── plan.md
        │   └── progress.json    # in-flight task progress (deleted on phase complete)
        └── archived-YYYYMMDD-phase-N/   # old plans, never deleted
```

## `state.json`

Canonical runtime state. Every command that mutates state updates this file. Schema:

```json
{
  "schema_version": 1,
  "mode": "from-scratch",
  "created": "2026-04-14T00:12:33Z",
  "current_phase": 2,
  "total_phases": 5,
  "last_session": "2026-04-14T03:47:01Z",
  "last_edit":    "2026-04-14T03:47:01Z",
  "last_commit":  "a1b2c3d"
}
```

| Field | Written by | Purpose |
|-------|-----------|---------|
| `schema_version` | `/sea-init` | Integer. Bump when state-shape changes; readers run migrations if older. |
| `mode` | `/sea-init` | `from-scratch` or `finish-existing` — affects /sea-go pipeline defaults |
| `created` | `/sea-init` | ISO-8601 UTC timestamp of initialization |
| `current_phase` | `/sea-go`, `/sea-roadmap` | Number of the active phase |
| `total_phases` | `/sea-init`, `/sea-roadmap` | Total phases in the roadmap |
| `last_session` | `/sea-go` end, `SessionStart` | Last time a command ran |
| `last_edit` | `PostToolUse` hook (`state-tracker`) | Last time any file was edited in this project by Claude |
| `last_commit` | `/sea-go` end, `/sea-quick` end | Short SHA of the most recent software-engineer-agent commit |

## `roadmap.md`

Human-readable phase list. The planner writes it, the user can hand-edit it, and `/sea-roadmap` edits it programmatically. Format:

```markdown
# Project Roadmap

## Project: <name>
## Created: <ISO date>
## Status: in-progress

## Phases

### Phase 1: <short name>
**Goal:** <one sentence>
**Scope:** <bullet list>
**Deliverable:** <what you end with>
**Depends on:** none | Phase X
**Status:** done | in-progress | pending

### Phase 2: ...
```

`Status` transitions: `pending` → `in-progress` → `done`. Once a phase is `done`, the `/sea-roadmap` skill refuses to remove or reorder it.

## `phases/phase-N/plan.md`

One plan file per phase, written by the `planner` agent in Mode B (Phase Planning) and consumed by the `executor`. Format:

```markdown
# Phase N Plan: <name>

## Context
<2-3 sentences>

## Complexity
trivial | medium | complex

## Pipeline
<describes which agents will be used>

## Tasks

### Task 1: <short name>
- **What:** <one sentence>
- **Files:** path1, path2 (new | modified)
- **Steps:**
  1. ...
  2. ...
- **Verification:** <exact command, expected output>
- **Commit:** `type(scope): message`

### Task 2: ...
```

If the planner cannot decide something, it inserts `[[ ASK: question ]]` markers. The `/sea-go` skill stops and surfaces these to the user before calling the executor.

## `phases/phase-N/progress.json` (transient)

Written by the executor after each successful task commit so the phase can resume after a crash, context reset, or `STATUS: blocked` exit.

```json
{
  "phase": 2,
  "current_task": 3,
  "completed_tasks": [1, 2],
  "last_commit": "a1b2c3d",
  "updated": "2026-04-14T05:21:00Z"
}
```

`/sea-go` checks this file before launching the executor:
- **Present** → resume context: skip `completed_tasks`, restart at `current_task`.
- **Absent** → fresh start at task 1.

Deleted automatically when the phase completes — `summary.md` becomes the historical record.

## `phases/phase-N/summary.md`

Written after the phase completes successfully. Short, appendable, human-readable:

```markdown
# Phase N Summary

Completed: <ISO now>
Commits: <count>  (<first-sha>..<last-sha>)
Files touched: <list>

Notes:
<2-3 sentences on what shipped, any deviations from the plan, anything the user should know>
```

## `diagnose.json`

Latest output from `/sea-diagnose`. Schema:

```json
{
  "generated": "2026-04-14T05:10:00Z",
  "focus": "all",
  "tests":    { "status": "warn", "findings": ["..."] },
  "errors":   { "status": "pass", "findings": [] },
  "security": { "status": "fail", "findings": ["..."] },
  "priority_actions": [
    "Add rate limiting to POST /api/login (src/routes/auth.ts:42)",
    "..."
  ]
}
```

Each section's `status` is one of `pass`, `warn`, `fail`. `/sea-status` reads this file to show the last audit result in its header.

## `.needs-verify` (transient)

A marker file created by `/sea-go` and `/sea-quick` after the executor finishes. Contains a single integer: the retry count (starts at `0`).

The `Stop` hook (`hooks/auto-qa`) checks for this file when Claude finishes responding:

1. No marker → hook exits 0, Claude stops normally
2. Marker exists → hook runs `scripts/detect-test.sh`, then runs the detected test command
3. Pass → marker is deleted, Claude stops
4. Fail → marker's counter is incremented, hook returns `{"decision": "block", "reason": "..."}`, Claude auto-retries the fix
5. After 2 failed retries → marker is deleted, hook returns a block with a "give up, report to user" reason

This file is transient: never commit it, never rely on its existence, never edit it by hand.

## `.last-verify.log` (transient)

Raw stdout + stderr of the last test run, written by `hooks/auto-qa`. Overwritten every run. Useful for debugging when auto-QA reports a failure — you can open it directly instead of re-running tests manually.

## Cross-Session Memory (NOT in `.sea/`)

Each subagent has its own `MEMORY.md` managed by the platform, stored under `.claude/agent-memory/<agent-name>/MEMORY.md`. This is distinct from `.sea/`:

| Location | Lifetime | Scope | Contents |
|----------|---------|-------|----------|
| `.sea/` | Project | Per-project runtime state | Roadmap, phase plans, current state |
| `.claude/agent-memory/<agent>/` | Agent, per project | Accumulated across sessions | Patterns, decisions, known pitfalls |

The platform loads the first 25KB of each `MEMORY.md` into the corresponding subagent at startup — we do not hand-manage this.

## Gitignore

`/sea-init` appends the following to `.gitignore` if it isn't already there:

```
# software-engineer-agent runtime state
.sea/
```

`.claude/agent-memory/` is NOT gitignored by default — agent learnings can legitimately be shared via git (they never contain secrets if the agent follows its rules). Decide per project whether to commit them.
